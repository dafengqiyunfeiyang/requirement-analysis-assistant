# Examples

## Example 1: Merchant-side signal with a likely valid direction

### Raw input

```text
销售说最近有几个商家在抱怨，说活动报名太麻烦，填很多东西，最后还不一定过。老板问我们是不是可以把报名流程简化一下，不然影响商家参与积极性。
```

### Good first-turn output shape

```text
当前梳理：
这条输入目前更像一个商家侧流程体验问题的需求线索。表面诉求是“简化活动报名流程”，但当前还不能直接写成需求，因为还不清楚问题到底出在步骤过多、审核规则不透明，还是活动门槛本身不合理。

用户价值：
目前能看到的用户价值主要偏商家侧。
可能的价值是降低报名理解成本和操作成本，减少放弃报名的情况。

业务价值：
可能相关的业务价值包括活动参与率提升、商家活跃提升。
但当前业务价值仍偏弱，因为还没有看到报名流失或活动参与转化方面的证据。

支持证据：
- 已有多个商家反馈报名麻烦
- 当前输入明确提到商家参与积极性受影响

反向证据 / 疑点：
- 还不清楚这是流程问题还是审核规则问题
- 还不清楚是普遍现象还是少量重点商家反馈

待补信息：
- 商家具体卡在哪一步
- 商家抱怨的是“操作复杂”还是“结果不透明”
- 当前是否有报名流失或退出数据

建议性倾向：
目前更像一个值得继续判断的商家侧需求线索，但还不建议直接把它写成“简化流程”的正式需求。

下一步建议：
1. 先补 2 到 3 个商家真实卡点案例
2. 再确认当前报名流程的关键流失位置
```

## Example 2: Consumer-side weak signal with solution-shaped feedback

### Raw input

```text
有用户说首页看不到想买的东西，建议把首页改得更丰富一点，多放点推荐坑位。
```

### Good first-turn output shape

```text
当前梳理：
这条输入目前更像一个方案化表达的反馈，还不是清晰需求。表面建议是“首页更丰富、多加推荐坑位”，但背后真正问题可能是找货效率低、内容匹配差，或首页信息组织不清。

用户价值：
当前用户价值还不明确。
“看不到想买的东西”可能意味着消费者发现效率低，但还没有足够信息确认问题具体发生在哪种场景。

业务价值：
可能相关的业务价值包括转化提升。
但目前业务价值很弱，因为还没有看到首页点击、跳失、找货效率、推荐效果等依据。

支持证据：
- 有用户明确反馈“看不到想买的东西”

反向证据 / 疑点：
- 当前只有结论，没有场景
- “多放推荐坑位”是方案，不一定是问题本身
- 不清楚这是个别反馈还是普遍现象

待补信息：
- 用户是在找明确商品，还是逛首页时没有内容承接
- 当前问题发生在新用户、老用户，还是特定品类用户
- 是否已有首页相关行为数据能说明找货效率问题

建议性倾向：
目前更像一条尚未成形的消费者反馈，还不足以支持“首页改版”或“增加推荐坑位”这类正式需求判断。

下一步建议：
1. 先确认用户具体找什么、在哪个场景下找不到
2. 补首页相关行为数据或典型案例
```

## What these examples demonstrate

- Do not upgrade messy input into a full requirement too early.
- Separate problem statements from solution-shaped statements.
- Use missing information as the main lever for progress.
- Keep the leaning suggestive, not decisive.
