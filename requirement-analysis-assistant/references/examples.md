# Examples

## Example 1: Merchant-side signal that becomes spec-ready

### Raw input

```text
销售说最近有几个商家在抱怨，说活动报名太麻烦，填很多东西，最后还不一定过。老板问我们是不是可以把报名流程简化一下，不然影响商家参与积极性。
```

### Good first-turn output shape

```text
一句话当前判断：
这条输入目前更像一个商家侧报名体验问题的需求线索，值得继续判断，但还不建议直接把“简化流程”写成正式需求。

用户：
- 主要受影响用户：商家
- 发起者：销售转述 + 老板关注
- 受影响者与发起者不完全一致，仍需补商家一手事实

场景：
- 发生在商家报名活动的流程中
- 关键节点可能在资料填写、提交审核、结果反馈

需求解释：
这条输入本质上不是在要“少几个字段”，而是在表达：商家报名活动时，理解成本和结果预期可能过高，影响继续参与。

新增判断：
- 当前最可能的问题不只是“步骤多”，也可能是“审核规则和结果反馈不清”
- 这条输入更像商家参与转化问题，而不是单纯页面表单问题

待确认信息：
- 商家具体卡在哪一步
- 商家更不满的是“操作复杂”还是“结果不透明”
- 当前是否已有报名流失或退出数据

冲突点：
- 暂无明显冲突点

已确认信息：
- [PM转述] 已有多个商家表达报名麻烦
- [PM转述] 老板已把它当成需要判断的主题

风险提示：
- 暂未识别明显法务/财务风险信号

是否建议转 spec：
暂不建议立即转 spec。先补商家真实卡点和流失证据，再判断是否已具备进入 spec 的基础。

用户价值：
当前可见的用户价值主要偏商家侧，可能是降低理解和操作成本，减少中途放弃。

业务价值：
可能相关的业务价值包括活动参与率提升、商家活跃提升，但目前证据还偏弱。

支持证据：
- 已有多个商家反馈
- 输入明确提到参与积极性受影响

反向证据 / 疑点：
- 还不清楚是流程问题还是审核规则问题
- 还不清楚是普遍现象还是少量重点商家反馈
```

### After more facts arrive

```text
PM补充：
商家主要卡在资质材料反复提交，且不知道为什么被拒。最近两周有 18% 商家在报名页退出。运营也说大量答疑都集中在“我为什么没过”。
```

### Good handoff moment

```text
一句话当前判断：
这条需求已经比较接近一个可进入 spec 的商家侧报名优化主题。

新增判断：
- “简化流程”不一定是最准确的表达，更接近“提升审核反馈清晰度，减少重复提交与退出”
- 当前问题已经有商家体验和业务转化两侧支撑

待确认信息：
- 审核拒绝原因是否已有结构化分类
- 当前后台是否已存这些原因但未前台透出

冲突点：
- 暂无明显冲突点

已确认信息：
- [PM确认] 商家会因审核反馈不清而重复提交
- [PM确认] 最近两周报名页存在 18% 退出
- [PM确认] 运营答疑集中在审核结果不透明

风险提示：
- 可能涉及财务影响，建议与财务确认：若优化后报名量提升，是否会影响活动成本或补贴测算
- 暂未识别明显法务风险信号

转 spec 前自检：
- 已自动纠正 1 处表达偏差：将“简化流程”收敛为“提升审核反馈清晰度，减少重复提交与退出”
- 未发现阻塞转 spec 的明显问题

是否建议转 spec：
建议转 spec。这条需求的问题定义、价值方向和主要边界已经足够支撑第一版 spec 启动。

转接提示：
这条需求已具备进入 spec 的基础。是否转入 spec 工作流？
如果继续，请上传当前使用的 spec 模板；如有使用说明和示范样板，也建议一并上传。
我会基于当前需求判断结果、spec 启动包和你上传的模板材料，自动生成一版可审查的 spec 骨架。
```

## Example 2: Consumer-side weak signal that should not hand off

### Raw input

```text
有用户说首页看不到想买的东西，建议把首页改得更丰富一点，多放点推荐坑位。
```

### Good first-turn output shape

```text
一句话当前判断：
这条输入目前更像一个方案化表达的消费者反馈，还不足以直接进入 spec。

用户：
- 主要影响用户：消费者
- 发起者：用户反馈
- 当前无法确认受影响范围

场景：
- 可能发生在首页浏览或找货场景
- 但具体是“明确找货”还是“逛首页发现内容”还不清楚

需求解释：
这条输入本质上可能是在表达“首页无法有效承接用户找货或发现需求”，而不是天然等于“多放推荐坑位”。

新增判断：
- 输入里的“多放推荐坑位”更像方案，不是需求本体
- 当前最大问题是问题场景和用户行为证据都不足

待确认信息：
- 用户是在找明确商品，还是逛首页时无内容承接
- 当前问题发生在新用户、老用户，还是特定品类用户
- 是否已有首页点击、跳失、找货效率等相关数据

冲突点：
- 暂无明显冲突点

已确认信息：
- [PM转述] 有用户明确反馈“看不到想买的东西”

风险提示：
- 暂未识别明显法务/财务风险信号

转 spec 前自检：
- 仍有 2 个问题建议先修正后再转 spec：问题场景不清、缺少行为证据

是否建议转 spec：
暂不建议转 spec。先把场景、用户行为和问题证据补清，再决定是否进入 spec。

用户价值：
当前用户价值方向可能是提升找货效率或内容匹配，但还不足以写实。

业务价值：
可能涉及转化提升，但当前缺少行为证据。

支持证据：
- 已有一条明确用户反馈

反向证据 / 疑点：
- 只有结论，没有场景
- 只有方案，没有问题定义
- 没有行为数据支撑
```

## Example 3: PM asks to enter spec but does not upload the template

### PM message

```text
可以转 spec 了，继续。
```

### Good response shape

```text
这条需求已具备进入 spec 的基础。

转 spec 前自检：
- 未发现阻塞转 spec 的明显问题

我可以继续转入 spec 工作流，但请先上传当前使用的 spec 模板。
如有使用说明和示范样板，也建议一并上传。

在你上传前，我先停在 spec 启动包，不自动脑补模板结构。

当前 spec 启动包摘要：
- 问题定义：……
- 用户：……
- 场景：……
- 需求解释：……
- 已确认信息：……
- 待确认信息：……
- 冲突点：……
- 风险提示：……
```

## What these examples demonstrate

- Do not upgrade messy input into a full requirement too early.
- Explain `用户`, `场景`, and `需求解释` before value analysis.
- Use `新增判断` as the highest-density section.
- Keep `待确认信息` and `已确认信息` short and purpose-driven.
- Keep risk prompts in-bounds.
- Use `转 spec 前自检` as the final guard before handoff.
- Do not enter spec drafting until the PM confirms handoff and uploads the current template materials.
