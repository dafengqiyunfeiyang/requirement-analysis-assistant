# 需求判断助手 v2 更新说明

本次更新基于试跑反馈，重点解决两类问题：

1. 输出有帮助，但信息冗余，PM 需要从大量已知信息里筛 20% 的有效新增内容
2. 需求判断结束后停得太早，没有顺滑进入 spec 的下一步

## 这次更新了什么

### 1. 先讲清 `用户 / 场景 / 需求解释`

在价值分析前，skill 现在会先解释：

- `用户`：这条输入主要影响谁、谁受益、谁在转述
- `场景`：问题发生在什么业务场景、链路节点和前提下
- `需求解释`：这条输入本质上要解决什么，而不是直接把方案表达当成需求

这样做是为了让后续的价值分析和 spec 转接不漂。

### 2. 输出改成“高密度状态化”

这次不是简单写短，而是减少重复阅读。

首屏现在优先输出：

- `一句话当前判断`
- `用户`
- `场景`
- `需求解释`
- `新增判断`
- `待确认信息`
- `冲突点`
- `已确认信息`
- `风险提示`
- `是否建议转 spec`

其中：

- `新增判断` 只保留 AI 这轮真正新增的分析价值
- `待确认信息` 通常控制在 1 到 3 条
- `已确认信息` 放在冲突点后，减少 PM 重复阅读已知背景

### 3. 新增风险提示

现在会在合适的时候提示：

- 法务风险方向
- 财务风险方向

边界是：

- 只提示“可能涉及什么”和“建议找谁确认”
- 不替法务或财务下结论
- 没有明显信号时，只会一句话带过，不强行铺满

### 4. 新增 spec 转接

当需求判断已经足够清楚时，skill 会自动询问是否转入 spec。

如果 PM 明确说：

- `转 spec`
- `写 spec`
- `可以进 spec`

则会直接进入 spec 转接流程。

### 5. 新增模板上传约束

进入 spec 后，AI 不会直接脑补模板结构。

现在会要求 PM 先上传当前使用的 spec 模板材料：

- 最低要求：当前母模板
- 推荐补充：使用说明、示范样板

在模板未上传前，assistant 会停在 `spec 启动包`，不会继续写 spec。

### 6. 新增 `spec 启动包`

在正式进入 spec 前，assistant 会先整理一版转接骨架，通常包括：

- 问题定义
- 用户
- 场景
- 需求解释
- 用户价值 / 业务价值
- 已确认信息
- 待确认信息
- 冲突点
- 风险提示
- 为什么现在适合进入 spec

在模板上传后，AI 会继续自动起草骨架层 spec，PM 主要做审查，不需要逐字段手填。

## 为什么这样改

- 为什么：解决“有用但太冗余”和“有用但停得太早”这两类核心反馈
- 风险：如果没有模板上传约束，AI 很容易把旧记忆或推测写成新模板规则
- 下一步：继续观察组员试跑时，对 `新增判断`、`是否建议转 spec` 和 `spec 启动包` 的接受度

## 对现有使用方式的影响

- skill 名称不变，仍然是 `requirement-analysis-assistant`
- 安装方式不变
- 旧调用方式仍可用
- 但如果希望体验到 v2 的完整能力，建议在调用时明确说明：
  - 先解释用户、场景和需求含义
  - 再判断是否具备转 spec 的基础

推荐调用示例：

```text
请使用 $requirement-analysis-assistant，先读取当前 project 的已有材料，先解释用户、场景和需求含义，再帮我判断这条散乱输入是否具备转 spec 的基础。
```
