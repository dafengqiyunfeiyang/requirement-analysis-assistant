# 需求判断助手 v2.1 更新说明

这次是一个补丁版本更新，重点只补一件事：

**在询问“是否转 spec”之前，assistant 先做一轮自我检查。**

## 为什么要加这一层

在 v2 里，需求判断已经能顺滑衔接 spec，但仍然存在一个风险：

- assistant 可能在问题定义还没完全收敛时，就过早询问是否转 spec

这会把仍有偏差的内容带进 spec，增加后续返工。

## v2.1 新增了什么

### 1. 新增 `转 spec 前自检`

现在在准备询问是否转 spec 之前，assistant 会先做一轮自检。

这轮自检主要检查：

- 问题定义是否被方案表达带偏
- `用户 / 场景 / 需求解释` 是否互相闭合
- 用户价值和业务价值是否过于空泛
- 是否还有会阻塞第一版 spec 的关键信息缺口
- 是否存在尚未处理的关键冲突
- 是否把推测写成了事实
- 是否漏掉了法务或财务风险信号

### 2. 自检采用“双层机制”

不是每轮都把检查结果大段展示给 PM。

而是：

- 日常轮次：内部静默自检
- 临近 spec 转接：显式输出 `转 spec 前自检`

这样做是为了既能纠错，又不把输出重新做重。

### 3. 自检结果只暴露高信号结论

对 PM 的外显结果只保留三类：

- `未发现阻塞转 spec 的明显问题`
- `已自动纠正 1-2 处表达偏差`
- `仍有 1-3 个问题建议先修正后再转 spec`

不会输出一大段 checklist。

### 4. 转 spec 的门槛更稳了

现在除了原有条件之外，还增加了一条：

- `转 spec 前自检` 不得发现明显阻塞问题

也就是说：

- 不是“信息差不多够了就问要不要转 spec”
- 而是“先自检，确认没有明显逻辑问题，再问要不要转 spec”

## 对使用方式的影响

- skill 名称不变
- 安装方式不变
- v2 的调用方式仍可用
- 但现在如果输入已经接近 spec 阶段，assistant 会先做一次更稳的转接前自检

推荐调用示例：

```text
请使用 $requirement-analysis-assistant，先读取当前 project 的已有材料，先解释用户、场景和需求含义，再在询问是否转 spec 前做一轮自检。
```

## 这次更新解决了什么

- 为什么：减少“过早转 spec”的风险
- 风险：如果把自检完整外显，会重新带来冗余输出
- 这次的控制方式：只暴露高密度结论，不暴露整段检查清单
